# Aruvi · Competency Mapping Constitution
## Subject Group: mathematics
## Stages: Middle Stage – Secondary Stage
## Task: chapter_to_competency_mapping
## Fetch Key: competency_constitution_mathematics
## Source: Aruvi_Competency_Mapping_Constitutions_V1.1 · Section III

*Exercise progression architecture · Built target vs infrastructure distinction · Single-chapter isolation*

---

**Rule 1.** The system SHALL conduct a primary inquiry evaluating the chapter's exercise and application architecture in absolute isolation to answer two sequential structural questions. First, the system MUST identify the distinct new cognitive mathematical operation that the graded exercise set explicitly builds through progressive steps. Second, the system MUST identify prior mathematical operations that the exercises require the student to execute as foundational tools, confirmed structurally by their presence in exercises without corresponding instruction, worked examples, or graded progression within this chapter.

**Rule 2.** To establish a structural match between the identified new cognitive operation and a C-code, the system SHALL strictly prohibit keyword or semantic matching. The system MUST map the cognitive operation's peak complexity — as demonstrated in the chapter's most advanced exploratory or application problems — to the specific stage-level demands of the Curricular Goals document. The system WILL confirm the match only if the chapter's peak exercise complexity structurally aligns with the C-code's explicit mathematical requirements for that stage, ensuring no confusion with superficially similar C-codes from other stages.

**Rule 3.** The system MUST execute the following pass/fail structural weight tests based solely on the chapter's internal exercise architecture, strictly treating concept introductions and worked examples as scaffolding only — not as evidence of weight:
- **Weight 3:** assigned if and only if the chapter's application and exploratory problems at peak complexity cannot be attempted without this competency as the primary cognitive operation — the competency is not a tool used to reach the answer but is itself the operation the problem is designed to exercise.
- **Weight 2:** assigned if and only if the exercise sets provide a dedicated graded progression for this competency that builds it distinctly, confirming it is genuinely built alongside the Weight 3 competency rather than scaffolding toward it.
- **Weight 1:** assigned if and only if the competency is introduced or developed only within isolated exercises or brief activities, confirming its presence but proving it is not a primary or substantive building target.
- **Infrastructure threshold:** zero weight when an operation is required to solve the chapter's exercises but receives no direct instruction or developmental progression within this chapter — if exercises demand flawless execution from the outset merely to access the new material, that operation is infrastructure regardless of its origin.

**Rule 4.** In the event two C-codes appear to satisfy the Weight 3 test simultaneously, the system MUST first re-examine whether one competency is reducible to a tool for the other at peak exercise complexity. If one is reducible, it receives Weight 2 and the other retains Weight 3. If both remain irreducible — neither can be classified as a tool for the other — both receive Weight 2 and no Weight 3 is assigned for this chapter.

**Rule 5.** The system MUST apply the Mathematics Boundary Rule to distinguish built targets from infrastructural tools using only this chapter's graded exercise progression. The system SHALL classify a mathematical operation as a built target only if this chapter's exercises provide explicit step-by-step graded developmental progression for it. If the exercises demand the operation's execution from the outset without instruction or progression, the system MUST classify it as infrastructure and assign zero weight. This determination requires no knowledge of curriculum sequence — it is made entirely from within this chapter's exercise architecture.

**Rule 6.** The system MUST strictly prohibit the following documents:
- **Learning Outcomes** — grade-bound milestones that bypass the structural analysis of exercise architecture required to determine mathematical competency targets
- **Pedagogy documents** — instructional strategies provide no observable structural evidence of the cognitive operations demanded by the chapter's exercise sets
- **Syllabus documents** — sequential topic lists introduce curriculum sequence knowledge that violates the constraint of mapping each chapter in complete isolation
- **Position Papers** — mathematical philosophy cannot serve as a pass/fail structural weight test within a specific chapter's exercise architecture
- **Assessment Framework documents** — summative evaluation design is structurally distinct from identifying what a chapter's exercise architecture is built to develop
