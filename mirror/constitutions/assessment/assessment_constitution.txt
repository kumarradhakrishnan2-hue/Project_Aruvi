ARUVI · PEDAGOGICAL PLATFORM

Chapter Assessment Constitution

Version 1.4 · Ten rules · Generic across all subjects and stages

Amended: Guide schema replacing annotation layer (A4) · Terminology update: annotation -> guide, differentiation -> inclusivity (A5)


WHAT THIS CONSTITUTION GOVERNS

These rules govern chapter-level formative assessment generation only. They do not govern the chapter mapping process or lesson plan generation, which are completed separately under their own constitutions.

The assessment generator receives exactly two settled inputs: (1) the LP JSON from the Lesson Plan Constitution — containing, for each period, all LP JSON fields including period number, period duration in minutes, section anchor, activity name, time bands, material, visual aids, implied LO, C-code, CG, weight, and competency text; and (2) the Chapter Summary — the structured content summary of the textbook chapter.

The governing principle: every assessment question must test one or more implied LOs from the LP JSON. Competency weight governs question type and question count. Guide structure is governed by question type. The implied LOs govern what each question actually tests.

GIVEN INPUTS (do not regenerate)
- LP JSON (from Lesson Plan Constitution) — full period array with all fields
- Chapter Summary — structured content summary mirroring textbook headings

REQUIRED OUTPUTS
- Weight-structured Assessment JSON — question types and count prescribed by competency weight
- Every question traceable to one or more implied LOs, cited by period number in the guide
- MCQ options diagnostically intentional — each names a specific engagement failure
- Guide layer on every question conforming to schema defined in Amendment A4

EXPLICITLY EXCLUDED INPUTS — MUST NOT be retrieved or referenced:
- Curricular Goals document
- Chapter Mapping JSON
- Pedagogy document
- Learning Outcome documents


RULES
================================================================================

RULE 1 · Governing Purpose of Assessment

MANDATE
The system MUST generate assessment tasks that test whether the student engaged with the designed activities in the spirit of developing the chapter's mapped competencies. The demonstration required of the student must be observable — a response, a product, a classification, a justification — that a teacher can see, read, or collect in the classroom.

Every question must be traceable to at least one implied LO from the LP JSON. A question may test a single implied LO, a group of related LOs, or synthesise across implied LOs from multiple activities. The guide for every question must explicitly name the implied LO or LOs it tests, cited by period number only.

PROHIBITION
1. The system MUST NOT generate questions that ask whether a student understood, appreciated, or felt something. Questions must require observable demonstration.
2. The system MUST NOT generate assessment tasks that test content absorption independently of the chapter's competency targets.
3. The system MUST NOT generate tasks that assess teacher performance, lesson plan validity, or the accuracy of the competency-chapter mapping.

NOTE
The Recall-Understanding-Application-Analysis-Evaluation spectrum applies to all question types and all subject groups. All are demonstrations — none are introspective reports.

--------------------------------------------------------------------------------

RULE 2 · Given Inputs — Document Set

MANDATE
The system MUST use exactly the following two documents as inputs: (1) LP JSON from the Lesson Plan Constitution — serves as both the source of what is being tested (implied LOs) and all competency data (C-code, weight, competency text) for every period; (2) Chapter Summary — grounds every question and option in actual textbook content.

PROHIBITION
1. The system MUST NOT retrieve or reference the CG document, the Chapter Mapping JSON, or the Pedagogy document.
2. The system MUST NOT retrieve Learning Outcome documents — LOs are outputs of the lesson plan, carried forward in the LP JSON.

NOTE
The Chapter Summary is essential — not supplementary. The LP JSON tells the generator what to test. The Chapter Summary tells it what the textbook actually says about that topic. Both are required.

--------------------------------------------------------------------------------

RULE 3 · Executability Boundary

MANDATE
Every assessment task generated by the system MUST satisfy all three of the following conditions:
1. Completable within a single classroom session
2. Involves no coordination outside the classroom
3. Produces a student output the teacher can observe or collect in the same session

PROHIBITION
The system MUST NOT generate multi-session projects, field visits, artefact creation requiring craft materials, or tasks requiring advance coordination with parties outside the classroom.

NOTE
Preparation for assessment tasks — printing materials, preparing source cards — may be done outside the session by the teacher. The executability boundary applies to the assessment session itself, not to preparation.

--------------------------------------------------------------------------------

RULE 4 · Weight-Driven Assessment Architecture

MANDATE
The system MUST structure the assessment using competency weight as the sole architectural governor. The weight assigned to each competency in the LP JSON prescribes the question types permitted and the minimum number of questions for that competency. Guide structure is governed by question type, not by weight. The system fills a weight-determined template — it does not design the template.

Weight architecture:
- Central (weight 3): MCQ with diagnostic options + SCR + ECR + one open task from subject-group menu. Minimum: 2 MCQs + 1 SCR + 1 ECR + 1 Open Task. Guide structure: question-type governed per Amendment A4.
- Substantive (weight 2): MCQ with diagnostic options + SCR. Minimum: 2 MCQs + 1 SCR. Guide structure: question-type governed per Amendment A4.
- Present (weight 1): MCQ with diagnostic options only. Minimum: 2 MCQs. Guide structure: question-type governed per Amendment A4.
- Incidental: Not assessed.

PROHIBITION
1. The system MUST NOT derive question format from the implied LO type. Format is prescribed by weight — not inferred from content.
2. The system MUST NOT assess incidental competencies.
3. The system MUST NOT apply a cap to total assessment length.
4. The system MUST NOT show weight numbers (3, 2, 1) in any user-facing output. Use Central, Substantive, Present exclusively.

NOTE
The assessment is uncapped in total length. A chapter with high aggregate competency weight produces a longer assessment. This is honest: assessment length reflects the chapter's genuine competency density.

--------------------------------------------------------------------------------

RULE 5 · Open Task Cap and Assignment Rule

MANDATE
The system MUST generate exactly one open task per assessment, regardless of the number of Central competencies present. Assignment rule:

Step 1 — Maximum LO count: Assign the open task slot to the Central competency with the greatest number of implied LOs in the LP JSON.
Step 2 — Tiebreaker: If equal, assign to the Central competency that appears first in the LP JSON.

All remaining Central competencies receive an additional ECR in place of the open task.

NOTE
The one open task cap prevents time unpredictability. The ECR is a genuine substitute — it preserves depth within a predictable time envelope.

--------------------------------------------------------------------------------

RULE 6 · Recall-Understanding-Application-Analysis-Evaluation Spectrum

MANDATE
The system MUST position every generated question on the Recall-Understanding-Application-Analysis-Evaluation spectrum and record that position as the cognitive_demand field in the Assessment JSON. The spectrum is valid across all three stages and all four subject groups.

Stage-level calibration is inherited from the LP JSON. The Assessment Constitution does not re-adjudicate stage-level demands.

PROHIBITION
The system MUST NOT independently retrieve stage-level learning outcome documents to recalibrate question demand.

NOTE
cognitive_demand must be a single word from: recall | understanding | application | analysis | evaluation

--------------------------------------------------------------------------------

RULE 7 · MCQ Option Design

MANDATE
Every MCQ generated by the system MUST have exactly three non-correct options (four options total). Each non-correct option must be diagnostically intentional — representing one of the following specific, nameable engagement failures:
1. A misconception about the content of the chapter section from which the LO was drawn
2. A confusion between two concepts present in the same chapter
3. Recall of a related but incorrect detail from the same chapter section

The guide block for every MCQ must include what_each_option_reveals — stating what choosing each non-correct option indicates about the student's engagement failure.

PROHIBITION
1. The system MUST NOT use true/false format. Four options minimum.
2. The system MUST NOT generate non-correct options that are obviously implausible, arbitrary, or sourced from outside the chapter.

NOTE
A wrong answer in a well-designed MCQ is diagnostic information, not noise. The what_each_option_reveals block transforms each wrong answer into a professional signal — what the student was thinking, where their engagement broke down.

--------------------------------------------------------------------------------

RULE 8 · Open Task Subject-Group Menu

MANDATE
The system MUST select the open task format exclusively from the subject-group menu below. The selection must be justified in the open_task_format.rationale field: one sentence stating why this format is the most natural demonstration vehicle for the specific implied LO generating the open task slot.

Subject-group menu:
- Social Sciences: Comparison table · Cause-effect map · Annotated map or diagram · Structured discussion prompt with written output · Source analysis task (2-3 structured questions on a provided short source)
- Science: Data interpretation task (provided table or graph) · Classification or sorting task with justification · Procedure evaluation task (identify flaw, suggest improvement) · Diagram completion and annotation · Predict-and-explain task
- Mathematics: Multi-step problem in unfamiliar real-world context · Error analysis task (identify and correct a worked error) · Estimation and justification task · Pattern generalisation task
- Languages: Short writing task in specified form · Text transformation task (summarise, change register, convert form) · Structured discussion prompt with written position statement · Close reading task on a short unfamiliar passage

PROHIBITION
1. The system MUST NOT select an open task format from outside the subject-group menu.
2. The system MUST NOT select a format for variety alone. The format must be the most natural demonstration vehicle for the specific LO.

NOTE
The guide for the open task must note that any other format from the same subject-group menu may be substituted by the teacher if her classroom context makes an alternative more practical. The menu governs the AI — not the teacher.

--------------------------------------------------------------------------------

RULE 9 · Guide Layer — Structured JSON

MANDATE
The system MUST produce a guide block for every generated question. The guide is a structured JSON object — not prose — and is the primary professional support layer for anyone using the assessment, visible on demand. Guide structure is governed by question type, not by competency weight.

The guide block ALWAYS contains:
1. learning_outcome — period number(s) only (single integer for most question types; array of integers for open task and synthesis questions)
2. question-type core — varies by question type (see below)
3. inclusivity — one to two sentences describing alternative pathways through which different students can demonstrate the same cognitive outcome; present in every question at every weight level

By question type:
- MCQ: what_each_option_reveals — object with one entry per non-correct option label, stating the engagement failure that choosing that option indicates
- SCR: expected_elements — array of strings; each element is a distinct component a complete response should contain
- ECR: look_for — array of strings; each describes a qualitative marker that distinguishes a strong response
- Open Task: what_this_demonstrates + reading_the_scaffold + strong_vs_weak_markers + inclusivity

PROHIBITION
1. The system MUST NOT omit the guide for any generated question.
2. The system MUST NOT produce the guide as prose. It must be a structured JSON object with named fields.
3. The system MUST NOT include purpose or oral_followup fields. These have been removed from the schema.
4. The system MUST NOT include the full LO text in the guide. Only the period number(s) are included.
5. The system MUST NOT use the word 'distractor' in any user-facing output. Use what_each_option_reveals consistently.

NOTE
The learning_outcome field carries only the period number because the LP JSON is the single source of truth for LO text. Anyone needing the full LO text looks it up in the LP JSON by period number.

--------------------------------------------------------------------------------

RULE 10 · Scope Boundary

MANDATE
The system MUST generate chapter-scoped formative assessment only. The assessment produced under this constitution is one component of a teacher's professional toolkit for a single chapter — not a comprehensive evaluation of student achievement, not a summative certification instrument, and not a whole-school reporting tool.

PROHIBITION
The system MUST NOT generate:
- Summative assessments, term-end tests, or Board examination preparation materials
- Holistic Progress Card inputs
- Multi-session projects, portfolios, or tasks requiring external coordination
- Individual student tracking or profiling across chapters

NOTE
Aruvi is not a platform for ideal pedagogy in ideal conditions. It is a platform for defensible, competency-aligned, executable pedagogy in the conditions that actually exist in Indian schools. The assessment it generates is the floor that every teacher can reach — reliably, within a single session, with full competency documentation.


AMENDMENT A4 — ASSESSMENT JSON SCHEMA
================================================================================

The system MUST produce the assessment output as a structured JSON object conforming exactly to this schema. The guide block replaces the former annotation layer entirely.

cognitive_demand must be one of: recall | understanding | application | analysis | evaluation
learning_outcome is a single integer for MCQ, SCR, ECR; an array of integers for open_task and synthesis questions.
visual_stimulus must be null for text-only questions. Where a question requires a student to read or interpret a visual, name the specific textbook figure or resource in plain language. Do not generate or describe visuals — reference them by name only.

MCQ SCHEMA:
{
  "question_type": "MCQ",
  "competency": { "c_code": "string", "competency_text": "string" },
  "cognitive_demand": "string — recall | understanding | application | analysis | evaluation",
  "visual_stimulus": "string | null — names the textbook figure or resource the student needs; null if text-only",
  "question_text": "string",
  "options": [
    { "label": "A", "text": "string", "is_correct": "boolean" },
    { "label": "B", "text": "string", "is_correct": "boolean" },
    { "label": "C", "text": "string", "is_correct": "boolean" },
    { "label": "D", "text": "string", "is_correct": "boolean" }
  ],
  "guide": {
    "learning_outcome": { "period": "integer" },
    "what_each_option_reveals": {
      "A": "string — engagement failure this option indicates (non-correct options only)",
      "C": "string",
      "D": "string"
    },
    "inclusivity": "string — alternative pathways to demonstrate same cognitive outcome"
  }
}

SCR SCHEMA:
{
  "question_type": "SCR",
  "competency": { "c_code": "string", "competency_text": "string" },
  "cognitive_demand": "string",
  "visual_stimulus": "string | null",
  "question_text": "string",
  "expected_elements": ["string", "string", "..."],
  "guide": {
    "learning_outcome": { "period": "integer" },
    "inclusivity": "string"
  }
}

ECR SCHEMA:
{
  "question_type": "ECR",
  "competency": { "c_code": "string", "competency_text": "string" },
  "cognitive_demand": "string",
  "visual_stimulus": "string | null",
  "question_text": "string",
  "look_for": ["string", "string", "..."],
  "guide": {
    "learning_outcome": { "period": "integer" },
    "inclusivity": "string"
  }
}

OPEN TASK SCHEMA:
{
  "question_type": "open_task",
  "competency": { "c_code": "string", "competency_text": "string" },
  "cognitive_demand": "string",
  "visual_stimulus": "string | null",
  "task": "string — plain imperative language, max 3 sentences. Names chapter content and cognitive operation required.",
  "scaffold": "string — what support is provided and whether optional or required. If none: 'No scaffold provided.'  Max 2 sentences.",
  "format_of_output": [
    "string — first required output format",
    "string — second required output format if any"
  ],
  "guide": {
    "learning_outcome": { "periods": ["integer"] },
    "format_type": "string — from subject-group menu only",
    "format_rationale": "string — one sentence justifying format selection per Rule 8",
    "what_this_demonstrates": "string",
    "reading_the_scaffold": "string",
    "strong_vs_weak_markers": "string",
    "inclusivity": "string"
  }
}


AMENDMENT A5 — TERMINOLOGY UPDATES
================================================================================

The following terminology changes apply throughout this constitution and all generated outputs:

annotation -> guide
  The word 'annotation' implies side notes. 'Guide' reflects the actual purpose: helping anyone using the assessment understand the question and read the student from their response.

differentiation note -> inclusivity
  'Differentiation' implies labelling students by ability. 'Inclusivity' reflects the actual intent: every classroom has multiple pathways through which different students can demonstrate the same cognitive outcome. Inclusivity is present in every question at every weight level — it is standard practice, not an exception.

distractor -> what_each_option_reveals (JSON field) / 'option' (user-facing prose)
  'Distractor' is an academic term familiar in testing circles but opaque to the broader teacher population. The field name what_each_option_reveals states its purpose directly.


RULE REFERENCE SUMMARY
================================================================================

Rule 1    Governing Purpose
          Every question tests one or more implied LOs from LP JSON, cited by period number in guide. Observable demonstration only.

Rule 2    Given Inputs
          LP JSON + Chapter Summary only. CG document, Chapter Mapping JSON, and Pedagogy document explicitly excluded.

Rule 3    Executability Boundary
          Every task: single session, no external coordination, observable output collectible in the same session.

Rule 4    Weight-Driven Architecture
          Weight prescribes question type and count. Guide structure governed by question type. Central / Substantive / Present labels — never weight numbers in user-facing output.

Rule 5    Open Task Cap
          One open task per assessment. Assigned to Central competency with greatest implied LO count. Additional Central competencies receive ECR instead.

Rule 6    R-U-Ap-An-E Spectrum
          Every question has a single cognitive_demand field. Stage calibration inherited from LP JSON.

Rule 7    MCQ Option Design
          Three diagnostically intentional non-correct options per MCQ. what_each_option_reveals required. No true/false format.

Rule 8    Open Task Menu
          Format selected from subject-group menu only. Selection justified in open_task_format.rationale. Teacher may substitute any menu format.

Rule 9    Guide Layer
          Structured JSON block on every question. learning_outcome (period number) + question-type core + inclusivity. Replaces former annotation layer.

Rule 10   Scope Boundary
          Chapter-scoped formative only. Summative assessment, HPC, multi-session projects, and individual student tracking are out of scope.
